# Feature: Editor image composition (single sticker)

Goal
Let a logged-in user ask the server to compose their current editor base image (temporary upload) with a single chosen sticker at a given position, producing a composed image file, without yet persisting it to the database or exposing any gallery.

Behavior
When the user is on `/editor` and already has a valid temporary base image (from the upload step), they can select one sticker and specify where it should appear on the image (e.g. coordinates within the image). The client sends a request to a new endpoint (e.g. `POST /editor/compose`) with the base image reference (from the session or a hidden field), the sticker identifier, and the position coordinates. The server checks that the user is authenticated, validates that the base image file exists and belongs to the current session, validates that the sticker exists in the stickers directory, validates the coordinates, then uses PHP’s GD functions to load the base image and sticker and overlay the sticker at the requested position. On success, the server writes the resulting composed image to a temporary directory and returns a response that allows the user to see the composed image (e.g. by redirecting to its temporary URL or back to `/editor` with the composed image selected). On failure (missing/invalid inputs, missing files, GD errors), the server re-renders the editor view with an appropriate error message.

Constraints
Use only PHP standard library (GD) for image processing. Sticker PNG transparency (alpha channel) must be preserved when compositing so that only the visible parts of the sticker are drawn over the base image. Do not persist any data to the `images` database table in this unit; only temporary files in a writable directory such as `public/tmp/` are allowed. The server must not trust client-provided filenames or paths directly: the base image must come from the current session (e.g. `$_SESSION['editor_temp_image']`), and the sticker must be resolved against the known stickers list from `StickerService`. Coordinate inputs must be validated and, if necessary, clamped so that the sticker is fully or partially within the base image bounds. All failure modes (missing auth, missing base image, invalid sticker, invalid coordinates, GD failure) must produce a safe error without leaking filesystem paths.

 Success Criteria
 An authenticated user with a valid temporary base image can send a composition request with a chosen sticker and coordinates and receive a composed image file generated by the server, visible via a temporary URL or within the editor page. Invalid inputs or missing resources result in the editor being re-rendered with a clear error message and no composed image being produced. No database rows are created or modified as part of this feature, and the behavior is independent of any future persistence or gallery features.
 
 ## Implementation Plan
 1. Add route `POST /editor/compose` → `EditorController::compose()`.
 2. In `compose()`: require auth and presence of base image in session; read sticker identifier and coordinates from request and validate.
 3. Implement `ImageComposeService` to load base and sticker images with GD, overlay sticker at given coordinates, and write composed image to `public/tmp/`.
 4. In `compose()`: call `ImageComposeService`; on success store composed filename in session or prepare it for the view; on failure collect error messages.
 5. Update `editor.php` to submit sticker choice and coordinates to `/editor/compose` and to display either the composed image or composition error messages.

## Tests

**Test cases**

- **Success:** Logged-in user with a valid temp base image in session POSTs to `/editor/compose` with valid sticker (e.g. `glasses.png`) and valid x, y → redirect or 200 with composed image available (e.g. new temp file or session key).
- **Failure:** POST `/editor/compose` without session → 302 to `/login`. Session present but no base image in session → editor re-rendered with composition error. Invalid or unknown sticker identifier → error. Invalid coordinates (e.g. negative, non-numeric) → error.
- **Edge:** Coordinates at or beyond image bounds (sticker partially off canvas) → accept if implementation clamps; otherwise defined behavior. Valid sticker filename that does not exist on server → error.

**Test Setup (authentication)**

```bash
BASE=http://localhost:8080

# Register
curl -s -c cookies.txt -X POST "$BASE/register" \
  -d "email=compose@test.com&username=composeuser&password=Secret123!&confirm_password=Secret123!"

# Login
curl -s -c cookies.txt -b cookies.txt -X POST "$BASE/login" \
  -d "email=compose@test.com&password=Secret123!"

# Upload a base image so session has editor_temp_image (required for compose)
curl -s -c cookies.txt -b cookies.txt -X POST "$BASE/editor/upload" \
  -F "base_image=@public/stickers/glasses.png"
```

**Execute tests**

```bash
BASE=http://localhost:8080

# Success: compose with valid sticker and coordinates (run after Test Setup)
curl -s -o /dev/null -w "%{http_code}" -b cookies.txt -X POST "$BASE/editor/compose" \
  -d "sticker=glasses.png&x=50&y=50"
# Expect: 302 or 200; then GET /editor or /tmp/<composed> shows composed image

# Failure: no session
curl -s -o /dev/null -w "%{http_code}" -X POST "$BASE/editor/compose" \
  -d "sticker=glasses.png&x=50&y=50"
# Expect: 302 to /login

# Failure: invalid sticker (not in allowed list)
curl -s -b cookies.txt -X POST "$BASE/editor/compose" \
  -d "sticker=../etc/passwd&x=0&y=0"
# Expect: 200 with editor page and composition error message

# Failure: missing coordinates or invalid (non-numeric)
curl -s -b cookies.txt -X POST "$BASE/editor/compose" \
  -d "sticker=glasses.png&x=abc&y=50"
# Expect: 200 with editor page and composition error
```
